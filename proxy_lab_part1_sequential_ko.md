# 4부 1장: 순차적 웹 프록시 구현

첫 번째 단계는 HTTP/1.0 GET 요청을 처리하는 기본적인 순차형 프록시를 구현하는 것이다. POST와 같은 다른 요청 타입은 엄격히 말해 선택 사항이다.

프록시가 시작되면, 명령줄에서 지정한 포트 번호로 들어오는 연결을 기다려야 한다. 연결이 성립되면, 프록시는 클라이언트로부터 요청 전체를 읽고 이를 파싱해야 한다. 그리고 클라이언트가 올바른 HTTP 요청을 보냈는지 판단해야 한다. 요청이 올바르다면, 프록시는 적절한 웹 서버와 자체적으로 연결을 맺은 뒤 클라이언트가 지정한 객체를 요청할 수 있다. 마지막으로, 프록시는 서버의 응답을 읽어 클라이언트에게 그대로 전달해야 한다.

## 4.1 HTTP/1.0 GET 요청

최종 사용자가 웹 브라우저의 주소창에 `http://www.cmu.edu/hub/index.html` 같은 URL을 입력하면, 브라우저는 프록시에게 다음과 비슷한 한 줄로 시작하는 HTTP 요청을 보낸다.

```http
GET http://www.cmu.edu/hub/index.html HTTP/1.1
```

이 경우 프록시는 최소한 다음 필드들을 파싱해야 한다.

- 호스트 이름: `www.cmu.edu`
- 경로 또는 쿼리와 그 뒤에 오는 모든 내용: `/hub/index.html`

이렇게 해야 프록시는 `www.cmu.edu`에 연결해야 한다는 사실을 알 수 있고, 그 서버로 다음과 같은 형식의 자체 HTTP 요청을 보낼 수 있다.

```http
GET /hub/index.html HTTP/1.0
```

HTTP 요청의 모든 줄은 캐리지 리턴 `\r` 뒤에 개행 문자 `\n`이 오는 형태로 끝난다는 점에 주의하라. 또 한 가지 중요한 점은, 모든 HTTP 요청은 빈 줄 하나로 종료된다는 것이다.

```text
"\r\n"
```

위 예제에서 웹 브라우저의 요청 줄은 `HTTP/1.1`로 끝나지만, 프록시의 요청 줄은 `HTTP/1.0`으로 끝난다는 점을 확인할 수 있다. 현대의 웹 브라우저는 HTTP/1.1 요청을 생성하지만, 여러분의 프록시는 이를 처리한 뒤 HTTP/1.0 요청으로 바꾸어 전달해야 한다.

HTTP 요청은, 단지 HTTP/1.0 GET 요청의 일부만 보더라도, 굉장히 복잡할 수 있다는 점을 고려하는 것이 중요하다. 교재는 HTTP 트랜잭션의 몇 가지 세부 사항만 설명하지만, 완전한 HTTP/1.0 명세는 RFC 1945를 참고해야 한다. 이상적으로는 여러분의 HTTP 요청 파서는 RFC 1945의 관련 부분에 따라 충분히 견고해야 한다. 단, 명세에서는 여러 줄에 걸친 요청 필드를 허용하지만, 여러분의 프록시는 이를 제대로 처리하지 않아도 된다. 물론 잘못된 요청이 들어왔다고 해서 프록시가 성급하게 중단되어서는 안 된다.

## 4.2 요청 헤더

이 랩에서 중요한 요청 헤더는 `Host`, `User-Agent`, `Connection`, `Proxy-Connection`이다.

- **항상 `Host` 헤더를 보내야 한다.**
  비록 이런 동작이 엄밀한 의미에서 HTTP/1.0 명세에 의해 허용된 것은 아니지만, 특히 가상 호스팅을 사용하는 일부 웹 서버로부터 제대로 된 응답을 끌어내기 위해 필요하다.

  `Host` 헤더는 최종 서버의 호스트 이름을 나타낸다. 예를 들어 `http://www.cmu.edu/hub/index.html`에 접근할 때, 프록시는 다음 헤더를 보내게 된다.

  ```http
  Host: www.cmu.edu
  ```

  웹 브라우저가 자신의 HTTP 요청에 `Host` 헤더를 붙여 보내는 경우도 있다. 그런 경우 프록시는 브라우저가 보낸 것과 같은 `Host` 헤더를 사용해야 한다.

- **다음 `User-Agent` 헤더를 항상 보내도록 선택할 수 있다.**

  ```http
  User-Agent: Mozilla/5.0 (X11; Linux x86_64; rv:10.0.3) Gecko/20120305 Firefox/10.0.3
  ```

  이 헤더는 과제 설명문에서 한 줄에 다 들어가지 않아 두 줄로 나누어 제시되었지만, 여러분의 프록시는 이를 한 줄로 보내야 한다.

  `User-Agent` 헤더는 운영체제나 브라우저 같은 매개변수 측면에서 클라이언트를 식별한다. 웹 서버는 이런 식별 정보를 바탕으로 자신이 제공하는 콘텐츠를 조정하는 경우가 많다. 간단한 텔넷 스타일 테스트를 할 때, 이 특정 `User-Agent` 문자열을 보내면 더 나은 내용과 더 다양한 자료를 돌려받는 데 도움이 될 수 있다.

- **항상 다음 `Connection` 헤더를 보내야 한다.**

  ```http
  Connection: close
  ```

- **항상 다음 `Proxy-Connection` 헤더를 보내야 한다.**

  ```http
  Proxy-Connection: close
  ```

  `Connection`과 `Proxy-Connection` 헤더는 첫 번째 요청/응답 교환이 끝난 뒤 연결을 계속 유지할지 여부를 지정하는 데 사용된다. 요청마다 프록시가 새로운 연결을 여는 것은 완전히 허용되며, 오히려 권장된다. 이 헤더들의 값으로 `close`를 지정하면, 프록시가 첫 번째 요청/응답 교환이 끝난 뒤 연결을 닫을 것이라는 점을 웹 서버에 알리게 된다.

편의를 위해, 위에서 설명한 `User-Agent` 헤더 값은 `proxy.c`에 문자열 상수로 제공된다.

마지막으로, 브라우저가 HTTP 요청의 일부로 추가적인 요청 헤더를 보낸다면, 프록시는 그것들을 변경하지 않고 그대로 전달해야 한다.

## 4.3 포트 번호

이 랩에서 중요한 포트 번호는 크게 두 가지 부류가 있다.

- HTTP 요청 포트
- 프록시의 리스닝 포트

HTTP 요청 포트는 HTTP 요청 URL 안에 선택적으로 포함될 수 있는 필드다. 즉, URL이 `http://www.cmu.edu:8080/hub/index.html` 형태일 수도 있다. 이 경우 프록시는 기본 HTTP 포트인 80이 아니라 포트 8080으로 `www.cmu.edu` 호스트에 연결해야 한다. URL에 포트 번호가 포함되어 있든 없든, 프록시는 올바르게 동작해야 한다.

리스닝 포트는 프록시가 들어오는 연결을 기다리는 포트다. 프록시는 자신의 리스닝 포트 번호를 지정하는 명령줄 인자를 받아들여야 한다. 예를 들어 다음 명령으로 실행하면, 프록시는 포트 15213에서 연결을 받아야 한다.

```bash
linux> ./proxy 15213
```

리스닝 포트는 비특권 포트(non-privileged port) 중에서 선택하면 된다. 즉, **1024보다 크고 65536보다 작은 포트**이면서, 다른 프로세스가 사용 중이지 않은 포트여야 한다. 각 프록시는 고유한 리스닝 포트를 사용해야 하고, 한 머신에서 많은 사람들이 동시에 작업할 수 있으므로, `port-for-user.pl` 스크립트가 제공된다. 이 스크립트를 사용하면 사용자 ID를 바탕으로 개인용 포트 번호를 생성할 수 있다.

```bash
linux> ./port-for-user.pl droh
droh: 45806
```

`port-for-user.pl`이 반환하는 포트 `p`는 항상 짝수다. 따라서 Tiny 서버처럼 추가 포트가 필요하다면, `p`와 `p + 1`을 안전하게 사용할 수 있다.

임의의 포트를 직접 고르지 말라. 그렇게 하면 다른 사용자의 작업에 간섭할 위험이 있다.
