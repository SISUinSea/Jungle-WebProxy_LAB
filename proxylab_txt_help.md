# Proxy Lab 안내문 한국어 번역

## 7. 평가

이 과제는 총 70점 만점으로 채점됩니다.

- **BasicCorrectness**: 기본 프록시 동작 구현 — 40점 (자동 채점)
- **Concurrency**: 동시 요청 처리 — 15점 (자동 채점)
- **Cache**: 동작하는 캐시 구현 — 15점 (자동 채점)

### 7.1 자동 채점

배포 자료에는 `driver.sh`라는 오토그레이더가 포함되어 있으며, 강사는 이를 사용해 **BasicCorrectness**, **Concurrency**, **Cache** 점수를 매깁니다. `proxylab-handout` 디렉터리에서 다음과 같이 실행합니다.

```bash
linux> ./driver.sh
```

이 드라이버는 반드시 **Linux 머신**에서 실행해야 합니다.

### 7.2 견고성 (Robustness)

늘 그렇듯, 오류뿐 아니라 잘못되었거나 악의적인 입력에도 견고하게 동작하는 프로그램을 제출해야 합니다.

서버는 일반적으로 오래 실행되는 프로세스이며, 웹 프록시도 예외가 아닙니다. 오래 실행되는 프로세스가 다양한 종류의 오류에 어떻게 반응해야 하는지 신중히 생각해야 합니다. 많은 종류의 오류에 대해, 프록시가 즉시 종료되는 것은 분명 적절하지 않습니다.

견고성은 그 외의 요구사항도 포함합니다. 예를 들면, segmentation fault 같은 오류 상황에 취약하지 않아야 하며, **메모리 누수**와 **파일 디스크립터 누수**도 없어야 합니다.

## 8. 테스트와 디버깅

간단한 오토그레이더 외에는, 구현을 테스트할 수 있는 예제 입력이나 테스트 프로그램이 따로 제공되지 않습니다. 따라서 스스로 테스트를 만들어야 하며, 필요하다면 디버깅과 구현의 정당성 판단을 돕기 위한 **테스트 하네스**까지 직접 만들어야 합니다. 이는 실제 환경에서 매우 가치 있는 능력입니다. 현실에서는 정확한 운영 조건이 드러나지 않는 경우가 많고, 참고할 수 있는 정답 구현도 없는 경우가 많기 때문입니다.

다행히 프록시를 디버깅하고 테스트하는 데 사용할 수 있는 도구는 많습니다. 모든 코드 경로를 충분히 실행해 보고, **기본 사례**, **일반적인 사례**, **경계 사례**를 포함하는 대표적인 입력 집합을 테스트해야 합니다.

### 8.1 Tiny 웹 서버

핸드아웃 디렉터리에는 CS:APP의 **Tiny 웹 서버** 소스 코드가 들어 있습니다. `thttpd`만큼 강력하지는 않지만, CS:APP Tiny 웹 서버는 필요에 따라 쉽게 수정할 수 있습니다. 또한 프록시 코드의 합리적인 출발점이기도 합니다. 그리고 드라이버 코드가 페이지를 가져올 때 사용하는 서버이기도 합니다.

### 8.2 telnet

교재 **11.5.3절**에서 설명했듯이, `telnet`을 사용해 프록시에 연결을 열고 HTTP 요청을 보낼 수 있습니다.

### 8.3 curl

`curl`을 사용하면 자신의 프록시를 포함해 어떤 서버로든 HTTP 요청을 생성할 수 있습니다. 이는 매우 유용한 디버깅 도구입니다. 예를 들어, 프록시와 Tiny가 둘 다 로컬 머신에서 실행 중이고, Tiny는 **15213번 포트**, 프록시는 **15214번 포트**에서 수신 대기하고 있다면, 다음과 같은 `curl` 명령으로 프록시를 통해 Tiny의 페이지를 요청할 수 있습니다.

```bash
linux> curl -v --proxy http://localhost:15214 http://localhost:15213/home.html
```

### 8.4 netcat

`netcat`, 즉 `nc`는 다목적으로 활용할 수 있는 네트워크 유틸리티입니다. `telnet`처럼 서버에 연결을 여는 데 사용할 수 있습니다. 따라서 프록시가 `catshark`에서 **12345번 포트**로 실행 중이라고 가정하면, 다음과 같이 프록시를 수동으로 테스트할 수 있습니다.

```bash
sh> nc catshark.ics.cs.cmu.edu 12345
GET http://www.cmu.edu/hub/index.html HTTP/1.0
HTTP/1.1 200 OK
...
```

웹 서버에 연결할 수 있을 뿐 아니라, `netcat`은 서버 자체로도 동작할 수 있습니다. 다음 명령으로 `12345`번 포트에서 수신 대기하는 `netcat` 서버를 실행할 수 있습니다.

```bash
sh> nc -l 12345
```

이렇게 `netcat` 서버를 띄워 두면, 프록시를 통해 그 서버의 가짜 객체에 요청을 보내고, 프록시가 `netcat`에 실제로 어떤 요청을 보냈는지 정확히 확인할 수 있습니다.

### 8.5 웹 브라우저

최종적으로는 최신 버전의 **Mozilla Firefox**를 사용해 프록시를 테스트해야 합니다. **About Firefox**에 들어가면 브라우저가 자동으로 최신 버전으로 업데이트됩니다.

Firefox가 프록시와 함께 동작하도록 설정하려면 다음으로 이동하십시오.

```text
Preferences > Advanced > Network > Settings
```

실제 웹 브라우저를 통해 프록시가 동작하는 모습을 보면 꽤 흥미로울 것입니다. 비록 프록시의 기능은 제한적이지만, 프록시를 통해 웹사이트의 대부분을 탐색할 수 있다는 것을 알게 될 것입니다.

다만 중요한 주의점이 있습니다. 웹 브라우저로 **캐싱**을 테스트할 때는 매우 조심해야 합니다. 최신 웹 브라우저는 자체 캐시를 가지고 있으므로, 프록시의 캐시를 테스트하기 전에 반드시 브라우저 캐시를 꺼야 합니다.

```bash

```

**사이트별 지침**: 각 학생이 자신의 `proxylab-handin.tar` 해답 파일을 어떻게 제출해야 하는지 설명하는 문단을 여기에 삽입하십시오.

- 교재 **10장~12장**에는 시스템 수준 I/O, 네트워크 프로그래밍, HTTP 프로토콜, 동시성 프로그래밍에 관한 유용한 정보가 담겨 있습니다.

## 10. 힌트

- 교재 **10.11절**에서 설명했듯이, 소켓 입력과 출력에 표준 I/O 함수를 사용하는 것은 문제가 됩니다. 대신 핸드아웃 디렉터리의 `csapp.c` 파일에 제공된 **Robust I/O (RIO)** 패키지를 사용할 것을 권장합니다.
- `csapp.c`에 제공된 오류 처리 함수는 프록시에 적절하지 않습니다. 서버는 한 번 연결을 받기 시작하면 종료되면 안 되기 때문입니다. 이 함수들을 수정하거나 직접 작성해야 합니다.
- 핸드아웃 디렉터리의 파일은 원하는 방식으로 자유롭게 수정해도 됩니다. 예를 들어, 좋은 모듈성을 위해 캐시 함수를 `cache.c`와 `cache.h` 같은 파일에 라이브러리 형태로 구현할 수 있습니다. 물론 새 파일을 추가하면 제공된 `Makefile`도 함께 업데이트해야 합니다.
- CS:APP3e 본문 **964쪽의 Aside**에서 논의했듯이, 프록시는 `SIGPIPE` 신호를 무시해야 하며, `write` 연산이 `EPIPE` 오류를 반환할 때도 이를 우아하게 처리해야 합니다.
- 때로는 조기에 닫힌 소켓에서 바이트를 받기 위해 `read`를 호출하면, `read`가 `-1`을 반환하고 `errno`가 `ECONNRESET`으로 설정될 수 있습니다. 이 오류 때문에도 프록시가 종료되어서는 안 됩니다.
- 웹의 모든 콘텐츠가 ASCII 텍스트는 아니라는 점을 기억하십시오. 웹에는 이미지와 비디오처럼 **바이너리 데이터**도 매우 많습니다. 네트워크 I/O용 함수를 선택하고 사용할 때 이 점을 반드시 고려해야 합니다.
- 원래 요청이 **HTTP/1.1**이라 하더라도, **모든 요청은 HTTP/1.0으로 전달**하십시오.
