# BSD 소켓 학습 가이드

## 왜 `spec.md`에서 BSD 소켓을 학습 키워드로 넣어뒀을까?

`spec.md`의 과제 흐름은 아래 순서로 되어 있다.

`소켓 이해 -> 간단한 클라이언트/서버 -> tiny 웹서버 -> 프록시 서버`

이 순서를 보면 BSD 소켓은 그냥 네트워크 배경지식이 아니라, 이번 과제 전체를 관통하는 구현 인터페이스다. 이유는 다음과 같다.

1. Tiny 웹서버와 Proxy 서버는 모두 결국 소켓으로 연결을 받고, 읽고, 쓰는 프로그램이다.
2. HTTP는 애플리케이션 프로토콜이지만, 실제 전송은 TCP 소켓 위에서 이루어진다.
3. 프록시는 특히 소켓을 더 잘 이해해야 한다.
클라이언트의 연결을 받는 서버 역할과, 원 서버에 다시 붙는 클라이언트 역할을 동시에 하기 때문이다.
4. `file descriptor`, `DNS`, `IP`, `TCP`, `HTTP`가 따로 적혀 있는 것도 BSD 소켓이 이 개념들을 한 번에 묶는 접점이기 때문이다.
5. 이 저장소의 코드도 BSD 소켓 API를 직접 쓰거나, 그 위를 감싼 래퍼를 사용한다.
예: `Open_listenfd`, `Open_clientfd`, `Accept`, `Rio_*`

즉 이번 과제에서 BSD 소켓은 "네트워크 통신을 실제 코드로 바꾸는 최소 공통 언어"다.

## 이번 과제 기준으로 어디까지 공부해야 할까?

결론부터 말하면, **"TCP 기반 클라이언트/서버를 C로 설명하고, Echo 서버/Tiny/Proxy의 흐름을 코드 수준에서 따라갈 수 있는 정도"** 까지가 핵심 범위다.

이번 과제 기준 필수 범위:

1. 소켓이 무엇인지
2. 소켓이 왜 file descriptor처럼 다뤄지는지
3. TCP 연결형 소켓의 흐름
4. 서버와 클라이언트의 API 호출 순서
5. `bind`, `listen`, `accept`, `connect`의 역할 차이
6. `read/write` 또는 `Rio_readlineb/Rio_writen`으로 데이터를 주고받는 방식
7. DNS 이름이 어떻게 IP로 해석되는지 (`getaddrinfo` 수준)
8. HTTP 요청/응답이 결국 소켓 위의 바이트 스트림이라는 점
9. 웹서버와 프록시에서 소켓이 각각 어떻게 쓰이는지

이번 과제 기준 있으면 좋은 범위:

1. `sockaddr`, `sockaddr_in`, `sockaddr_storage`
2. `SO_REUSEADDR`가 왜 필요한지
3. blocking I/O와 동시성의 관계
4. 클라이언트 소켓 하나, 리스닝 소켓 하나, 연결 소켓 하나의 구분
5. 포트 번호와 잘 알려진 포트, ephemeral port 개념

이번 과제 기준 과하게 깊게 들어갈 필요는 없는 범위:

1. raw socket
2. 패킷 캡처/커널 네트워크 스택 내부 구현
3. `epoll`, `kqueue`, `select` 최적화
4. TLS/HTTPS 내부 구현
5. UDP 멀티캐스트, 브로드캐스트

## 큰 범위

이번 과제에서 BSD 소켓 학습 범위는 크게 5개로 자르면 된다.

### 1. 소켓의 정체

- 소켓은 네트워크 통신의 끝점(endpoint)이다.
- 유닉스에서는 소켓도 파일 디스크립터로 다룬다.
- 그래서 `read`, `write`, `close` 같은 I/O 모델을 공유한다.
- 다만 일반 파일과 달리, 네트워크 반대편 프로세스와 연결된 커널 객체라는 점이 중요하다.

### 2. TCP/IP 위에서 소켓이 놓이는 위치

- 애플리케이션은 HTTP를 말한다.
- HTTP 메시지는 TCP 연결 위에서 바이트 스트림으로 전달된다.
- TCP는 IP 위에서 동작한다.
- BSD 소켓 API는 애플리케이션이 TCP/IP를 직접 다루지 않고 OS를 통해 사용하게 해 주는 인터페이스다.

핵심 한 줄:

`HTTP는 내용`, `TCP는 전달 방식`, `소켓은 그것을 코드로 다루는 창구`

### 3. 서버와 클라이언트의 생명주기

클라이언트:

`socket -> getaddrinfo -> connect -> read/write -> close`

서버:

`socket -> bind -> listen -> accept -> read/write -> close`

여기서 특히 구분해야 할 것:

- `listenfd`: 요청을 받기 위해 열어두는 소켓
- `connfd`: 실제로 특정 클라이언트와 통신하는 연결 소켓

`accept`는 기존 `listenfd` 자체를 통신용으로 바꾸는 게 아니라, **새 연결용 fd를 하나 더 만들어 반환**한다.

### 4. 주소와 이름

- 프로그램은 종종 `www.example.com` 같은 이름을 받는다.
- 실제 연결은 IP 주소와 포트 번호로 해야 한다.
- 그래서 `getaddrinfo`가 이름을 주소 후보들로 바꿔 준다.
- 이 과제에서는 `Open_clientfd`, `Open_listenfd`가 이 과정을 감싸고 있다.

### 5. 이번 과제와 직접 연결되는 사용 방식

- Echo 서버: 소켓으로 연결 받고 받은 바이트를 그대로 돌려준다.
- Tiny 서버: 요청 라인을 읽고, 파일이나 CGI 결과를 HTTP 응답으로 써 준다.
- Proxy 서버: 브라우저의 요청을 읽고, 원 서버에 새 소켓으로 연결해서 다시 요청한 뒤, 응답을 원래 클라이언트에 전달한다.

프록시가 어려운 이유는 소켓을 한 방향이 아니라 **양쪽에 걸쳐 중계**해야 하기 때문이다.

5. 클라이언트는 왜 `connect`를 하냐?

3. `accept`가 새 `connfd`를 반환하는 이유
4. 연결 후에는 양쪽 모두 읽기/쓰기를 할 수 있다는 점

### 목표 3. 코드와 연결

특히 아래 매핑을 이해해야 한다.

- `Rio_readlineb`, `Rio_writen`: 소켓 위에서 안전하게 읽고 쓰기

### 목표 4. 프록시 관점 설명

아래 문장을 자기 말로 설명할 수 있어야 한다.

`프록시는 클라이언트에게는 서버이고, 원 서버에게는 클라이언트다.`

즉 프록시는:

1. 브라우저의 연결을 받기 위해 서버 소켓이 필요하고
2. 원 서버에 접속하기 위해 클라이언트 소켓이 필요하다

## 발표/설명용 핵심 목차

남에게 설명할 때는 아래 순서가 가장 자연스럽다.

### 1. 소켓이 왜 필요한가

- 프로세스끼리 네트워크로 통신하려면 OS가 제공하는 인터페이스가 필요하다.
- 그 인터페이스가 BSD 소켓이다.

### 2. 소켓은 무엇인가

- 네트워크 통신의 끝점
- 유닉스에서는 fd로 표현되는 I/O 객체

### 3. TCP 서버/클라이언트는 어떻게 동작하는가

클라이언트:
`connect`로 서버에 붙는다.

서버:
`bind`로 주소를 점유하고, `listen`으로 요청을 받을 준비를 하고, `accept`로 실제 연결을 하나씩 꺼낸다.

### 4. 데이터는 어떻게 오가나

- 연결 후에는 양쪽이 바이트 스트림을 읽고 쓴다.
- HTTP도 결국 문자열 형태의 요청/응답을 바이트로 보내는 것이다.

### 5. 그래서 웹서버와 프록시가 어떻게 만들어지나

- 웹서버는 요청을 읽고 응답을 쓴다.
- 프록시는 요청을 읽고, 다른 서버에 다시 보내고, 응답을 받아 원래 클라이언트에 돌려준다.

## 꼭 설명할 수 있어야 하는 포인트

### 1. `socket()`과 `connect()`는 역할이 다르다

- `socket(domain, type, protocol)`은 "어떤 종류의 통신 수단을 만들지" 정한다.
- 예를 들면 IPv4/IPv6 계열인지, 스트림(TCP)인지 같은 통신 방식이다.
- 아직 누구와 통신할지는 정해지지 않는다.
- `connect()`는 "그 소켓을 어느 원격 주소와 연결할지" 정한다.

즉:

- `socket()`은 통신 도구 생성
- `connect()`는 상대 지정

서버 쪽에서는 `bind()`가 "내가 어느 로컬 주소/포트를 사용할지" 정한다.

### 2. `accept()`는 왜 새 fd를 반환하나

- 서버는 계속 새 요청을 받아야 한다.
- 그래서 듣는 역할의 `listenfd`는 남겨 두고,
- 실제 통신은 요청마다 별도의 `connfd`로 한다.

이 구분이 안 되면 Tiny와 Proxy 구조를 이해하기 어렵다.

### 3. 소켓은 "동기화된 파일"이라기보다 "커널이 관리하는 양끝 연결"이다

- 파일처럼 fd로 다루지만, 일반 파일과 완전히 같은 것은 아니다.
- 한쪽이 `write`한 데이터가 네트워크를 통해 반대편 `read`에 도착한다.
- 핵심은 "같은 파일을 공유"하는 게 아니라, **연결된 두 끝점 사이에 바이트 스트림이 흐른다**는 점이다.

## 권장 학습 순서

1. 소켓과 fd의 관계 이해
2. 클라이언트/서버 API 순서 암기
3. Echo 예제로 요청/응답 흐름 이해
4. Tiny에서 HTTP 요청 읽기와 응답 쓰기 이해
5. Proxy에서 "받고 -> 다시 연결하고 -> 전달" 흐름 이해
6. `csapp.c`의 `open_clientfd`, `open_listenfd` 구현 읽기

## 코드와 연결해서 보면 좋은 지점

### Echo 클라이언트

- `webproxy-lab/echo/echoclient.c`
- `Open_clientfd(host, port)`로 서버에 연결
- `Rio_writen`, `Rio_readlineb`로 송수신

### Echo 서버

- `webproxy-lab/echo/echoserveri.c`
- `Open_listenfd(argv[1])`로 리스닝 소켓 생성
- `Accept`로 연결 수락
- `echo(connfd)`로 통신

### Tiny

- `webproxy-lab/tiny/tiny.c`
- `doit(connfd)`에서 HTTP 요청 라인 읽기
- 파일 또는 CGI 결과를 HTTP 응답으로 작성

### Proxy

- `webproxy-lab/proxy.c`
- 클라이언트 요청을 받은 뒤
- 원 서버에 새 연결을 만들고
- 응답을 다시 원래 클라이언트에게 전달해야 함

## 이 정도면 충분한가?

아래를 할 수 있으면 이번 과제 기준으로 충분하다.

1. `socket/bind/listen/accept/connect`를 말로 설명할 수 있다.
2. Echo 서버와 Echo 클라이언트의 호출 순서를 코드 없이 써낼 수 있다.
3. `listenfd`와 `connfd`의 차이를 설명할 수 있다.
4. HTTP가 TCP 소켓 위의 텍스트 기반 프로토콜이라는 점을 설명할 수 있다.
5. 프록시가 소켓을 두 번 쓴다는 점을 설명할 수 있다.
6. `Open_clientfd`, `Open_listenfd`가 내부적으로 무엇을 감싸는지 설명할 수 있다.

## 한 문장 학습 목표

`BSD 소켓을 이용해 TCP 클라이언트/서버의 연결, 송수신, 주소 처리 흐름을 설명하고, Tiny 웹서버와 Proxy 서버 구현에 그 지식을 바로 연결할 수 있다.`

## 발표 직전 체크리스트

- 나는 소켓을 "네트워크 통신의 끝점"이라고 설명할 수 있는가?
- 나는 소켓이 fd로 다뤄지는 이유를 설명할 수 있는가?
- 나는 서버 쪽 호출 순서를 설명할 수 있는가?
- 나는 클라이언트 쪽 호출 순서를 설명할 수 있는가?
- 나는 `accept`가 왜 새 fd를 주는지 설명할 수 있는가?
- 나는 프록시가 왜 더 어렵고 왜 소켓 이해가 더 필요한지 설명할 수 있는가?
- 나는 HTTP 요청이 결국 소켓에 쓰는 바이트라는 점을 설명할 수 있는가?

이 체크리스트에 모두 답할 수 있으면, 이번 과제에서 요구하는 BSD 소켓 학습 목표는 충족했다고 봐도 된다.
