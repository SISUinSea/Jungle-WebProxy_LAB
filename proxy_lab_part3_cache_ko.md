# 6부: 웹 객체 캐싱

랩의 마지막 부분에서는, 최근에 사용된 웹 객체를 메모리에 저장하는 캐시를 프록시에 추가하게 된다. HTTP는 실제로 웹 서버가 자신이 제공하는 객체를 어떻게 캐시해야 하는지 지시를 내리고, 클라이언트가 캐시를 자신을 대신해 어떻게 사용해야 하는지 지정할 수 있도록 꽤 복잡한 모델을 정의하고 있다. 하지만 여러분의 프록시는 이를 단순화한 방식으로 채택할 것이다.

프록시가 서버로부터 웹 객체를 받으면, 그 객체를 클라이언트로 전송하는 동안 메모리에 캐시해야 한다. 이후 다른 클라이언트가 같은 서버에 있는 같은 객체를 요청하면, 프록시는 서버에 다시 연결할 필요 없이 캐시된 객체를 그대로 다시 보내면 된다.

물론 프록시가 지금까지 요청된 모든 객체를 전부 캐시한다면, 무한한 메모리가 필요하게 된다. 게다가 어떤 웹 객체는 다른 객체보다 더 크기 때문에, 하나의 매우 큰 객체가 캐시 전체를 차지해 다른 객체들이 전혀 캐시되지 못하는 상황도 생길 수 있다. 이러한 문제를 피하기 위해, 여러분의 프록시는 **최대 캐시 크기**와 **최대 캐시 객체 크기**를 모두 가져야 한다.

## 6.1 최대 캐시 크기

프록시 캐시 전체의 최대 크기는 다음과 같다.

```c
MAX_CACHE_SIZE = 1 MiB
```

캐시 크기를 계산할 때, 프록시는 **실제 웹 객체를 저장하는 데 사용되는 바이트만** 계산해야 한다. 메타데이터를 포함해 그 외의 부가적인 바이트는 모두 무시해야 한다.

## 6.2 최대 객체 크기

프록시는 다음 크기를 넘지 않는 웹 객체만 캐시해야 한다.

```c
MAX_OBJECT_SIZE = 100 KiB
```

편의를 위해, 두 크기 제한은 모두 `proxy.c`에 매크로로 제공되어 있다.

정확한 캐시를 구현하는 가장 쉬운 방법은 **각 활성 연결마다 버퍼를 하나씩 할당**하고, 서버로부터 데이터를 받을 때마다 그 버퍼에 누적하는 것이다. 만약 버퍼 크기가 어느 시점에서든 최대 객체 크기를 초과하면, 해당 버퍼는 버릴 수 있다. 반대로 웹 서버의 응답 전체를 최대 객체 크기를 넘기기 전에 모두 읽었다면, 그 객체는 캐시에 저장할 수 있다.

이 방식을 사용하면, 여러분의 프록시가 웹 객체 때문에 사용하게 되는 최대 데이터 양은 다음과 같다. 여기서 `T`는 활성 연결의 최대 개수이다.

```c
MAX_CACHE_SIZE + T * MAX_OBJECT_SIZE
```

## 6.3 축출 정책

프록시의 캐시는 **최소 최근 사용(Least Recently Used, LRU)** 축출 정책을 근사하는 정책을 사용해야 한다. 반드시 엄격한 LRU일 필요는 없지만, 그것에 충분히 가까운 방식이어야 한다. 또한, 객체를 **읽는 것**과 **쓰는 것** 모두 그 객체를 “사용한 것”으로 간주된다는 점에 유의하라.

## 6.4 동기화

캐시에 대한 접근은 반드시 **스레드 안전(thread-safe)** 해야 하며, 캐시 접근에서 경쟁 상태(race condition)가 발생하지 않도록 보장하는 것이 아마 이 파트에서 더 흥미로운 부분이 될 것이다. 실제로는 특별한 요구사항도 하나 있는데, **여러 스레드가 동시에 캐시를 읽을 수 있어야 한다.**

물론 한 번에 하나의 스레드만 캐시에 쓸 수 있어야 하지만, 이 제한이 읽는 스레드들에게까지 적용되어서는 안 된다.

따라서 캐시 전체를 하나의 큰 **배타적 락(exclusive lock)** 으로 보호하는 방식은 허용되지 않는다. 대신 캐시를 여러 부분으로 나누거나, Pthreads의 **readers-writers lock** 을 사용하거나, 혹은 세마포어를 사용해 직접 **readers-writers 문제**를 해결하는 방법 등을 고려해볼 수 있다. 어느 방식을 쓰든, 엄격한 LRU를 구현할 필요까지는 없다는 사실이 여러 리더를 지원하는 데 어느 정도 유연성을 제공해준다.
